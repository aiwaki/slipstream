# Decisions

Stable decisions and invariants for Slipstream. Add entries when a rule should
survive across sessions and agents.

| Date | Decision | Status |
|---|---|---|
| 2026-07-18 | The production-facing Windows service controller acquires the shared machine-wide operation lock before collecting durable lifecycle state or live SCM/ownership evidence and holds that same lock through reconstruction, reducer execution, and every native effect. Only a committed active-install record plus exact live owned identity becomes actionable. Fresh or terminal exact absence remains idempotent; reinstalling the identical already-running service is a no-op. Interrupted, inaccessible, inconsistent, foreign, or unknown evidence remains non-mutating. Disposable CI must prove a failed restart persists its bounded attempt and a later controller process resumes recovery before this boundary may feed a production host or any Windows networking effect. | Active |
| 2026-07-17 | The native Windows lifecycle is composed behind one shared operation-lock acquisition per action; filesystem and SCM sub-effects expose lock-free internal entry points only for that compositor. Final cleanup is ordered by evidence, not by request acceptance: unregister closes the exact service and manager handles, then waits boundedly until `dev.slipstream.service` is truly absent before the owned executable may be removed. A post-commit failure preserves the exact identity in the absent tombstone and defers active-install clearing until both exact SCM absence and exact payload-path absence are proven. Unknown, foreign, inaccessible, or conflicting evidence remains non-mutating. Disposable CI must run a real service through install, stop, start, crash recovery, uninstall, and post-commit compensation before any Windows networking effect is admitted. | Active |
| 2026-07-17 | Every privileged Windows service lifecycle, staged-payload, and SCM effect acquires the same machine-wide v1 operation lock before reading authorization evidence and holds it through commit, compensation, or native mutation. The lock is bounded, non-inheritable, and created with a LocalSystem/built-in Administrators DACL. Because Windows ignores supplied security attributes when a named object already exists, every returned kernel-object handle is independently checked for a trusted owner and allow-only trusted writer/waiter principals before use; squatted, wrong-type, inaccessible, or permissive objects fail closed. Abandoned ownership is safe only because each effect revalidates strict durable evidence before acting. An accepted stop request must keep the same exact service handle and wait to a bounded, identity-matching `Stopped` state before unregister may run. Timeout, unexpected state, or changed evidence fails closed. | Active |
| 2026-07-17 | Native Windows SCM mutation is isolated behind a pure action-specific v1 gate. Register requires exact absence, zero-attempt running intent, no active-install record, and exact staged payload. Start, stop, and unregister require compatible durable intent, a matching-or-missing active record, exact staged payload, and same-handle stable ownership proof; an absent install-rollback tombstone may authorize only cleanup. Each native action opens only `dev.slipstream.service` with its single mutation right plus query rights, never enumerates or reconfigures services, and contains no process, DNS, proxy, PAC, VPN, socket, or packet API. Foreign, transitional, unknown, or mismatched evidence performs no mutation. | Active |
| 2026-07-17 | Durable Windows service state is a separate filesystem-only transaction. Strict bounded v1 intent and active-install records live under the protected machine directory and use owner-only, exact non-reparse handles. A pending file is registered before write, flushed, verified from the same handle, and committed with same-volume write-through rename. Any pending, inaccessible, invalid, permissive, orphaned, or identity-mismatched evidence blocks future SCM mutation. Stable state is only a prerequisite for evaluating a separate action-specific ownership gate, never authorization by itself. Active-install commit requires matching zero-attempt running intent plus exact owned staged-payload evidence; clearing it requires and preserves an absent intent tombstone and verifies disappearance. The module contains no SCM, process-control, DNS, proxy, PAC, VPN, socket, or packet API. | Active |
| 2026-07-17 | Packaged Safari lifecycle cleanup treats both zombie state (`Z`) and macOS process-exit state (`E`) as terminal observations. Once `ps` reports a same-UID Safari PID as exiting, the harness waits for natural removal and never tries to re-identify or signal it from an unavailable `(Safari)` command line. All non-terminal process control still requires the exact Safari executable identity. | Active |
| 2026-07-17 | Native Windows payload staging is a separate filesystem-only effect for `StagePayload` and `RemoveOwnedPayload`. The source, pending executable, final executable, pending owner record, and final owner record are opened as non-reparse handles and bounded before use. The source SHA-256 must equal the lifecycle identity. New directories and files use a protected LocalSystem/built-in Administrators DACL; each pending handle is registered for compensation before fallible I/O, the executable is flushed and committed first, and the strict owner record is the final commit marker. Success requires staging-only read evidence to reopen both and reproduce the exact identity while the transaction still holds its write/delete handles; this only widens sharing, grants no write access, and does not loosen the normal ownership collector. Failure and removal mark only already-proven handles for deletion; foreign, permissive, ambiguous, or colliding paths are never replaced. This module has no SCM, process-control, DNS, proxy, PAC, VPN, socket, or packet API. | Active |
| 2026-07-17 | Windows ownership evidence is collected from handles, never trusted path strings. The v1 machine record lives under the system `FOLDERID_ProgramData` known folder with no environment-variable fallback. The collector requires an exact final DOS path, a non-reparse regular file, a LocalSystem or built-in Administrators owner, no write-capable allow ACE for any other principal, bounded strict JSON, and a same-handle executable SHA-256. Missing, inaccessible, malformed, complex, or permissive evidence remains non-owned. The collector has no write, service-control, process, DNS, proxy, VPN, socket, or packet API. | Active |
| 2026-07-17 | Windows service ownership is a separate pure v1 proof contract. `Owned` requires an owner-only machine record whose exact service name, canonical SCM command, absolute executable path, lowercase SHA-256, and positive generation agree with a stable read-only SCM observation and independently verified executable evidence. Missing, inaccessible, malformed, permissively writable, or transitional evidence is `Unknown`; path, command, name, or hash mismatch is `Foreign`. Neither JSON content nor service presence can claim ownership by itself. This contract performs no filesystem, ACL, process, SCM, DNS, proxy, VPN, socket, or packet effect. | Active |
| 2026-07-17 | Native Windows service discovery begins with a query-only SCM observer isolated from lifecycle decisions. It opens only the exact `dev.slipstream.service` name and may read status and configured binary command. Only `ERROR_SERVICE_DOES_NOT_EXIST` proves absence; access errors, malformed data, transitional states, and unknown values never become absent or owned. A present service remains unowned evidence until a separate owner-only record, expected path, executable hash, and generation agree. This observer cannot create, start, stop, delete, execute, or reconfigure a service and cannot touch DNS, proxy, PAC, VPN, sockets, or packets. | Active |
| 2026-07-17 | Windows service lifecycle v1 is a separate effect-injected contract before native service-manager code. Every durable intent is bound to the exact `dev.slipstream.service` identity, executable SHA-256, generation, and crash-attempt count. Install commits only after readiness and compensates to a verified absence. Stop and uninstall persist non-running intent before service actions; `absent` intent cannot be weakened by start or stop, and clearing the active install record must preserve that tombstone. Crash recovery runs only for exact owned state with `running` intent, persists the bounded attempt before restart, and resets it only after readiness. Cleanup may tolerate intermediate errors only when the final state is proven; failed compensation becomes unknown and never auto-recovers. Foreign or unknown services receive no destructive effect. | Active |
| 2026-07-17 | Windows is the first platform adapter, but its v1 boundary is deliberately no-network and effect-injected. It must execute the shared routing, recovery, StatusV2, signed-bundle, and activation contracts with deterministic fake effects before native service, process, filesystem, DNS, proxy, VPN, socket, packet, or installer code is admitted. Rollback durable commit and runtime activation are one explicit atomic effect: failure leaves the current policy active. Native lifecycle work must retain this harness as a regression boundary and qualify install, start, stop, crash recovery, and rollback on disposable Windows before routing effects are introduced. | Active |
| 2026-07-17 | Every Python runtime path that activates or rolls back a signed route policy goes through activation contract v1. The adapter executes reducer actions in order, keeps candidate and rollback file changes transactional, restores the prior runtime policy on effect failure, and records every consumed trial generation in an owner-only activation file before candidate activation or health checks. Successful commits mirror that generation in backward-compatible policy metadata. Persisted signed provenance remains explicit even when its content has the bundled SHA-256; a newly verified candidate whose canonical SHA-256 already equals the active policy remains the frozen v1 `no_change` case and creates no trial. Direct signed apply/save bypasses are removed. The remote channel remains disabled, no production key is added, and this migration changes no PF, DNS, proxy, PAC, VPN, or route classification behavior. | Active |
| 2026-07-17 | Signed route-policy activation contract v1 is a pure compare-and-swap reducer shared by Python and `slipstream-core`. Only an already-verified signed identity may enter one trial at a time. Trial start and rollback require the expected active SHA-256; health results require both the exact candidate SHA-256 and the reducer-issued monotonic trial generation, so an aborted attempt cannot authorize a retry of identical content. A candidate commits only after a completed gate with at least one success and zero degraded or blocked checks. Rejection restores the stable active policy, rollback consumes one previous slot before falling back to bundled policy, and adapter actions are ordered data only. This extraction adds no fetch, persistence, trust key, runtime apply, PF, DNS, proxy, PAC, VPN, or routing behavior. | Active |
| 2026-07-16 | Signed route-policy bundle contract v1 is shared by pure Python and `slipstream-core`. The fixture key is test-only: this extraction adds no production trust key, remote fetch, automatic apply, filesystem, or network behavior. | Active |
